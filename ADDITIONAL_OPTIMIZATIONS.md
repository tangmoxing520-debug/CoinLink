## 一、价格获取和数据处理优化

### 🔴 优化1：价格获取失败时的容错处理

**位置**：`trader_v2.py:990-993`, `trader_v2.py:1193-1195`

**问题**：
- `get_current_price()` 失败时直接返回 `None`，导致开仓/平仓检查失败
- 没有重试机制，网络波动可能导致误判

**建议**：
```python
# 增加重试机制
def get_current_price(self, symbol: str, max_retries: int = 3) -> Optional[float]:
    for attempt in range(max_retries):
        try:
            price = self.fetcher.get_current_price(symbol)
            if price and price > 0:
                return price
        except Exception as e:
            if attempt < max_retries - 1:
                time.sleep(1)
                continue
            logging.warning(f"获取 {symbol} 价格失败 (尝试 {attempt+1}/{max_retries}): {e}")
    return None
```

**优先级**：P1

### 🟡 优化2：K线数据缓存机制

**位置**：`trader_v2.py:951`, `live_trader_v3.py:460, 673`

**问题**：

- 在 `_scan_opportunities()` 和 `open_position()` 中重复获取

**建议**：

- 避免同一币种在短时间内重复请求

**优先级**：P2

---

### 🟡 优化3：K线数据时效性检查

**位置**：`trader_v2.py:951-954`

**问题**：
- 不检查K线数据的时间戳
- 可能使用过时的数据计算信号评分

**建议**：
```python
# 检查最新K线的时间戳
latest_timestamp = df.index[-1]
age_minutes = (datetime.now() - latest_timestamp).total_seconds() / 60
if age_minutes > 5:  # 数据超过5分钟认为过期
    logging.warning(f"{symbol} K线数据过期 ({age_minutes:.1f}分钟)")
    return None
```

**优先级**：P1

## 二、信号评分和一致性优化

### 🔴 优化4：信号评分计算与回测对齐

**位置**：`trader_v2.py:956-958`, `trader_v2.py:143-192`

**问题**：
- 实盘使用 `SignalScorerLive`（简化版）
- 回测使用 `SignalScorer`（完整版）
- 可能导致评分不一致

**建议**：
- 对比两个评分器的逻辑差异
- 确保实盘和回测使用相同的评分算法
- 或者：明确标注差异是设计选择

**优先级**：P1

---

### 🟡 优化5：信号评分缓存

**位置**：`trader_v2.py:956-958`

**问题**：
- 同一币种在同一时间窗口内可能被多次评分
- 浪费计算资源

**建议**：
- 缓存最近1分钟内的评分结果
- 键：`(symbol, trigger_change, coin_change, category)`

**优先级**：P2

## 三、订单执行和错误处理优化

### 🔴 优化6：实盘订单执行的重试机制

**位置**：`trader_v2.py:1100-1143`

**问题**：
- 下单失败后直接返回 `None`
- 没有区分临时错误（网络）和永久错误（余额不足）

**建议**：
```python
# 区分错误类型
if isinstance(e, BinanceAPIError):
    if e.code in [1003, -1021]:  # 临时错误：网络问题、时间戳错误
        # 可以重试
        if attempt < max_retries:
            time.sleep(2)
            continue
    elif e.code in [-2010, -2011]:  # 永久错误：余额不足、订单被拒绝
        # 不重试，直接返回
        return None
```

**优先级**：P1

---

### 🟡 优化7：订单状态确认增强

**位置**：`trader_v2.py:1118-1129`

**问题**：
- 只等待0.5秒确认订单
- 如果订单部分成交，可能创建不准确的持仓记录

**建议**：
- 增加轮询机制，最多等待5秒
- 如果订单未完全成交，记录警告并继续监控

**优先级**：P2

## 四、持仓监控和风险管理优化

### 🔴 优化8：持仓检查频率优化

**位置**：`live_trader_v3.py:297-310`

**问题**：
- 每个监控周期都检查所有持仓
- 如果持仓很多，可能影响扫描机会的效率

**建议**：
- 持仓检查可以独立线程或异步执行
- 或者：降低持仓检查频率（每2个周期检查一次）

**优先级**：P2

---

### 🟡 优化9：持仓价格更新失败的处理

**位置**：`trader_v2.py:1193-1195`

**问题**：
- 如果 `get_current_price()` 失败，持仓检查会跳过
- 可能导致止损/止盈延迟触发

**建议**：
- 记录价格获取失败的次数
- 连续失败3次后，使用最后一次成功价格（带警告）
- 或者：触发紧急平仓（如果价格获取失败超过5分钟）

**优先级**：P1

---

### 🟡 优化10：动态止损更新频率

**位置**：`trader_v2.py:1212`

**问题**：
- 每次持仓检查都更新移动止损
- 如果检查频率很高，可能过于频繁

**建议**：
- 移动止损更新可以限制频率（例如：每30秒最多更新一次）
- 或者：只在价格变化超过阈值时更新

**优先级**：P2

## 五、性能和资源优化

### 🟡 优化11：监控循环的性能优化

**位置**：`live_trader_v3.py:297-310`

**问题**：
- `_scan_opportunities()` 可能耗时较长
- 如果扫描时间超过监控间隔，可能导致延迟

**建议**：
- 记录每个步骤的耗时
- 如果总耗时超过监控间隔的50%，发出警告
- 考虑并行处理多个分类的扫描

**优先级**：P2

---

### 🟡 优化12：API请求速率限制优化

**位置**：`gate_data_fetcher.py:53-59`

**问题**：
- 固定速率限制（0.5秒）可能过于保守
- 在需要快速响应时可能延迟

**建议**：
- 实现动态速率限制
- 根据API响应时间调整请求间隔
- 或者：使用令牌桶算法

**优先级**：P2

## 六、数据一致性和同步优化

### 🔴 优化13：实时价格与K线数据同步

**位置**：`trader_v2.py:990`, `trader_v2.py:1193`

**问题**：
- `get_current_price()` 使用ticker数据
- `get_klines()` 使用K线数据
- 两者可能不同步，导致计算偏差

**建议**：
- 使用K线的最新收盘价作为当前价格（如果可用）
- 或者：记录价格来源，便于调试

**优先级**：P1

**位置**：`trader_v2.py:1082-1089`

**建议**：
- 定期（每小时）同步持仓数据
- 如果发现不一致，记录警告并尝试修复

**优先级**：P2
